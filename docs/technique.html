Introduction et Contexte Technique

Bienvenue dans la documentation technique du BotPromo Jinx. Ce document s'adresse aux développeurs, aux contributeurs potentiels, ainsi qu'aux étudiants curieux.

Choix technologiques et Architecture

Le projet repose sur plusieurs piliers techniques :

  • Python 3 & interactions.py : Le bot est entièrement codé en Python. L'utilisation de la bibliothèque moderne interactions.py permet une gestion native, asynchrone et optimisée des Slash Commands de Discord.
  • Architecture Modulaire (Cogs) : Le code est strictement séparé selon le principe de responsabilité unique. L'interface avec Discord et la déclaration des commandes sont isolées dans le dossier Cogs/ (ex: Salles.py), tandis que la logique métier et le parsing des données se trouvent dans des modules indépendants (ex: TrouveTaSalle.py).
  • Programmation Concurrente (Multithreading) : Pour éviter la latence lors de la mise à jour des emplois du temps, les lourdes requêtes HTTP vers les serveurs de l'IUT et l'analyse des calendriers sont déléguées à un pool de threads secondaires (via concurrent.futures.ThreadPoolExecutor).
  • Conteneurisation : Le bot est packagé avec Docker, garantissant un environnement d'exécution stable et indépendant de la machine hôte.

Moteur EDT (TrouveTaSalle.py)

Ce fichier contient le cœur logique du bot. Il s'occupe de récupérer et d'analyser les emplois du temps de l'IUT de Bayonne.

Récupération des données (ICS)

Le bot fait des requêtes GET vers l'URL https://www.iutbayonne.univ-pau.fr/outils/edt/default/export?ID=... pour télécharger les données au format standard iCalendar (.ics).

Algorithme de traitement

  1. Parallélisation : La méthode refresh() utilise un pool de threads pour télécharger les EDT de toutes les promos simultanément.
  2. Filtrage Temporel : Les événements passés de la journée ou ceux des jours suivants sont ignorés pour optimiser la mémoire.
  3. Dédoublonnage : Si plusieurs TPs ont cours en même temps dans la même salle, l'algorithme fusionne les événements.
  4. Détection des créneaux libres : La méthode detecter_creneaux_libres_salle() calcule les "trous" entre les cours de la journée (généralement entre 7h45 et 18h30) pour déterminer les disponibilités d'une salle.

Interface Discord (Cogs/Salles.py)

Cette extension charge les commandes liées aux salles et aux professeurs. Elle intègre des dictionnaires constants associant les rôles Discord à leurs IDs pour les semestres pairs et impairs.

Tâche de fond (Background Task)

Le module initialise un Thread détaché (Daemon) exécutant la fonction refresh_edt(). Ce thread appelle la méthode de mise à jour toutes les 10 minutes en journée. En dehors des heures de cours, le thread se met en sommeil prolongé pour économiser la bande passante.

Gestion des Salons

Une fonction utilitaire _warn_wrong_channel() vérifie l'ID du salon où la commande est exécutée. Si l'utilisateur n'est pas dans le bon salon, la commande est bloquée et un message éphémère d'avertissement est renvoyé.

Point d'entrée (main.py)

Le fichier principal orchestre l'initialisation du bot :

  • Chargement sécurisé du Token depuis les variables d'environnement via dotenv.
  • Définition des Intents Discord (limités au strict nécessaire : guild messages et membres).
  • Chargement dynamique de l'extension Cogs.Salles.
  • Définition de commandes utilitaires de base (/ping, /clear, /8ball).
