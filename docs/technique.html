Introduction et Contexte Technique

Bienvenue dans la documentation technique du BotPromo Jinx. Ce document s'adresse aux développeurs, aux contributeurs potentiels, ainsi qu'aux étudiants curieux.

Choix technologiques et Architecture

Le projet repose sur plusieurs piliers techniques :

  • Python 3 & interactions.py : Le bot est entièrement codé en Python. L'utilisation de la bibliothèque moderne interactions.py permet une gestion native, asynchrone et optimisée des Slash Commands de Discord.
  • Architecture Modulaire (Cogs) : Le code est strictement séparé selon le principe de responsabilité unique. L'interface avec Discord et la déclaration des commandes sont isolées dans le dossier Cogs/ (ex: Salles.py), tandis que la logique métier et le parsing des données se trouvent dans Cogs/src/ (ex: TrouveTaSalle.py).
  • Programmation Concurrente (Multithreading) : Pour éviter la latence lors de la mise à jour des emplois du temps, les lourdes requêtes HTTP vers les serveurs de l'IUT et l'analyse des calendriers sont déléguées à un pool de threads secondaires (via concurrent.futures.ThreadPoolExecutor).
  • Conteneurisation : Le bot est packagé avec Docker, garantissant un environnement d'exécution stable et indépendant de la machine hôte.

Moteur EDT

Ce fichier contient le cœur logique du bot. Il s'occupe de récupérer et d'analyser les emplois du temps de l'IUT de Bayonne.

Récupération des données (ICS)

Le bot fait des requêtes GET vers l'URL des emplois du temps de l'IUT pour télécharger les données au format standard iCalendar (.ics). Les salles sont divisées en deux catégories : listeSallesPC (salles avec ordinateurs) et listeSallesTD (salles classiques).

Algorithme de traitement

  1. Parallélisation : La méthode refresh() utilise un pool de threads pour télécharger les EDT de toutes les promos simultanément.
  2. Filtrage Temporel : Les événements passés de la journée ou ceux des jours suivants sont ignorés pour optimiser la mémoire et les performances.
  3. Dédoublonnage : Si plusieurs groupes ont cours en même temps dans la même salle (ex: CM partagé), l'algorithme fusionne intelligemment les événements.
  4. Détection des créneaux libres : La méthode detecter_creneaux_libres_salle() calcule les "trous" entre les cours de la journée (généralement entre 7h45 et 18h30) pour déterminer les disponibilités exactes d'une salle.
  5. Recherche de professeurs : La méthode get_prof() parcourt les descriptions des événements ICS pour localiser un professeur spécifique (matière, salle, horaire).

Interface Discord

Cette extension charge les commandes liées aux salles et aux professeurs. Elle gère également la configuration selon le semestre universitaire (pair/impair).

Configuration du Semestre (SEMESTER)

Une variable d'environnement SEMESTER dicte le dictionnaire ID_PROMOS qui est utilisé. Les identifiants (ID) des calendriers ICS changent entre les semestres impairs (S1/S3) et les semestres pairs (S2/S4).

Tâche de fond (Background Task)

Le module initialise un Thread détaché dans la fonction setup() exécutant refresh_edt(). Ce thread appelle la mise à jour des emplois du temps toutes les 10 minutes en journée. La nuit, le thread se met en sommeil prolongé pour économiser les requêtes (plusieurs heures).

Point d'entrée

Le fichier principal orchestre l'initialisation du bot :

  • Chargement sécurisé du Token depuis les variables d'environnement via dotenv ou un fichier confidentiel.py local.
  • Définition des Intents Discord (limités au strict nécessaire : messages du serveur et membres).
  • Chargement dynamique de l'extension principale Cogs.Salles.
  • Définition des commandes globales utilitaires de base (/ping, /clear, /8ball).
